# Adaptive Bandit with Simulated Annealing — Multi-Phase Implementation Plan

**Source:** Synergy E from `ideas_v2.md` (Risk 2/7, Reward 5/7)
**Status:** NOT IMPLEMENTED
**Depends on:**
- Model Scientist Pipeline (Synergy C) — **IMPLEMENTED** (`model_scientist/`)
- Surrogate Triage Pipeline (Synergy B) — **IMPLEMENTED** (`surrogate_triage/`)
- GPU Kernel Creation Pipeline (Synergy D) — **IMPLEMENTED** (`gpu_kernels/`)

**Goal:** Replace the current fixed-fraction allocation and greedy acceptance across all three systems with a unified Thompson Sampling bandit that controls *which type of modification* gets tried (arm selection across 9+ categories spanning Python-level, paper-sourced, and kernel-level proposals) and *whether to accept a risky result* (per-arm simulated annealing where temperature derives from posterior variance). The bandit unifies the three-way `EvaluationScheduler` split (internal 60% / paper 25% / kernel 15%), the `ColdStartManager` regime transitions, the `EvolutionaryRefinementScheduler` kernel selection, and the `FailureExtractor` modification categories into a single adaptive decision layer — one JSON state file, one selection logic, one acceptance criterion, continuously learning from every experiment in `hypothesis_journal.jsonl`.

### From Surrogate Triage Pipeline (`surrogate_triage/`)

| Component | File | How Bandit Pipeline Uses It |
|---|---|---|
| `EvaluationScheduler` | `funnel/evaluation_scheduler.py` | **Replaced** by the bandit: the fixed three-way split (internal 60% / paper 25% / kernel 15%) is replaced by Thompson Sampling across all arms. Paper-sourced and kernel-sourced candidates compete on equal footing with internal proposals — the bandit learns the optimal allocation from data. The `EvaluationScheduler` is retained only as a fallback during cold-start (< 50 journal entries) before the bandit has enough data |
| `QueueManager` | `funnel/queue_manager.py` | When the bandit selects a paper-sourced or kernel arm, it pops from the queue filtered by `modification_category` matching the selected arm. If the queue has no candidates matching the selected arm, the bandit resamples (up to 3 retries) before falling back to an internal proposal |
| `SurrogateScoringPipeline` | `funnel/scoring_pipeline.py` | The surrogate's `predicted_delta` is used as a prior for the bandit's acceptance decision: if the surrogate predicted the candidate would be bad (high adjusted_score, i.e., large predicted regression) AND the actual result is also bad, the annealing acceptance probability is further penalized because the regression was predictable, not exploratory. Conversely, if the surrogate predicted the candidate would be good but the result was bad, the annealing is more lenient — the mismatch suggests the arm's landscape has shifted |
| `SurrogateFeedbackLoop` | `funnel/feedback_loop.py` | Extended to record the bandit's arm selection and temperature alongside the surrogate's prediction error — this enables analysis of whether the bandit's arm preferences correlate with surrogate accuracy (e.g., "the surrogate is poorly calibrated for architecture proposals but well-calibrated for optimizer proposals") |
| `ColdStartManager` | `surrogate/cold_start.py` | Extended with bandit-aware regimes: `no_bandit` (< 30 entries, use `EvaluationScheduler` fallback), `conservative_bandit` (30-99 entries, use bandit but with minimum exploration rate of 20% per arm), `full_bandit` (100+ entries, unrestricted Thompson Sampling). The transition thresholds are higher than the surrogate's cold-start (50/200) because the bandit needs more diverse data to form meaningful posteriors |
| `PaperSourceTracker` | `funnel/source_tracker.py` | Per-arm success rates from `PaperSourceTracker` are used to warm-start the paper-sourced arms' posteriors — if `technique_category: "architecture"` has a 40% success rate across 25 evaluations in `paper_source_quality.json`, the paper-architecture arm starts with `Beta(10, 15)` rather than `Beta(1, 1)` |
| `KnowledgeCeilingMonitor` | `steady_state/ceiling_monitor.py` | The ceiling monitor's `paper_fraction` trend informs a meta-level bandit adjustment: if `paper_fraction` is rising (Python-level optimization saturating), the bandit's paper arms get a prior boost proportional to the trend magnitude. This is a soft coupling — the bandit can override this if paper arms start performing poorly |
| `ConstraintPreFilter` | `extraction/constraint_filter.py` | Constraint penalty from pre-filtering is incorporated into the bandit's temperature for that arm-proposal pair: `effective_temperature = T_arm × (1 + constraint_penalty)` — proposals with high constraint penalty need a higher temperature bar to be accepted via annealing because they're in known-dangerous territory |
| `BottleneckMapper` | `intelligence/bottleneck_mapper.py` | Bottleneck mappings inform arm boosting: when a bottleneck like `attention_entropy_collapse` is active, the `architecture` arm and the `paper_architecture` arm both get a diagnostics boost to their Beta prior (temporary additive shift to alpha, decaying over 20 iterations) |
| `IngestionBiasAgent` | `intelligence/ingestion_bias.py` | The source quality boosts computed by `IngestionBiasAgent` are accessible to the bandit for paper arms — a paper arm with high source quality boost gets a slightly higher prior mean, reflecting the meta-learning about which paper sources produce good candidates |

## Phase 2: Thompson Sampling Selection Engine

**Gate:** Phase 2 is COMPLETE when: (1) the `ThompsonSamplerEngine.select()` method produces arm selections that match the theoretical Thompson Sampling distribution over 1000 simulated iterations (chi-squared goodness-of-fit test, p > 0.05, verified by `SelectionDistributionValidator`); (2) the exploration floor is enforced: no arm is selected fewer than `exploration_floor × total_iterations` times in a 200-iteration simulation; (3) diagnostics-informed boosting correctly shifts selection probabilities: when a synthetic `DiagnosticsReport` with `attention_entropy < 0.1` is injected, the `architecture` arm's selection frequency increases by at least 10% compared to no diagnostics; (4) the prompt routing layer produces category-specific prompts that are verifiably different (cosine similarity between prompt embeddings for different arms < 0.8); (5) every selection event is logged to `bandit_log.jsonl` with all required fields and the log can reconstruct the exact selection sequence when replayed.

**Gate:** Phase 3 is COMPLETE when: (1) the acceptance pipeline correctly implements per-arm temperatures: improvements always accepted, regressions accepted with probability `exp(-delta / T_arm)`, verified by running 500 iterations and confirming that arms with higher temperatures accept more regressions than arms with lower temperatures (Spearman correlation > 0.7 between temperature and annealing acceptance rate); (2) at least 2 modifications are accepted via annealing (not by improvement) in a 100-iteration run, and at least 1 of those annealing-accepted modifications leads to a subsequent improvement within 10 iterations (validating the "stepping stone" mechanism); (3) reheating triggers correctly when an arm accumulates `K_reheat_threshold` consecutive failures (verified by injecting a synthetic failure streak and confirming temperature spikes by `reheat_factor`); (4) the rollback safety net correctly reverts an annealing acceptance that leads to 3 consecutive worse results; (5) the surrogate prediction modulates annealing acceptance: when the surrogate predicted a regression AND the result is a regression, the effective temperature is halved (reducing exploratory acceptance of predictably bad outcomes).

## Full Pipeline Integration

9. Delayed corrections from `RuntimeCorrectnessMonitor` have been correctly applied to at least 1 kernel arm posterior
10. All bandit-derived metrics are in the `MetricRegistry` with computed `correlation_with_success` values
