# GPU Kernel Creation Pipeline — Multi-Phase Implementation Plan

**Source:** Synergy D from `ideas_v2.md` (Risk 6/7, Reward 6/7)
**Status:** NOT IMPLEMENTED
**Depends on:**
- Model Scientist Pipeline (Synergy C) — **IMPLEMENTED** (`model_scientist/`)
- Surrogate Triage Pipeline (Synergy B) — **IMPLEMENTED** (`surrogate_triage/`)

**Goal:** Extend autoresearch below the Python/PyTorch abstraction layer to generate, verify, and evolve custom Triton kernels — using the Model Scientist's diagnostics to identify *where* kernel optimization matters, the Surrogate Triage's scoring infrastructure to predict *which* kernel proposals are worth testing, and both systems' journal/failure-mining/ablation pipelines to track results with the same scientific rigor as algorithmic changes. Kernel-level speedups stack multiplicatively with algorithmic gains.

## Phase 1: GPU Profiling Integration & Correctness Harness

**Gate:** Phase 1 is COMPLETE when: (1) `DiagnosticsReport` includes a `kernel_profiling` section with per-operation wall-clock time, memory bandwidth utilization (GB/s vs. theoretical peak), and SM occupancy for the top-15 most expensive operations in `train.py`, validated by `DiagnosticsValidator`; (2) the correctness harness classifies 3 known-correct Triton kernels as PASS and 2 intentionally-broken kernels (off-by-one indexing, wrong accumulation dtype) as FAIL with clear error messages identifying the exact tensor position and magnitude of divergence; (3) the benchmarking harness produces timing measurements with coefficient of variation < 5% across 5 runs and correctly identifies a known-faster fused elementwise kernel as faster than the PyTorch reference. Manual review of profiling data and correctness reports required.

### Subphase 1.1: Profiling Instrumentation Inside DiagnosticsInstrumenter

**Status:** NOT IMPLEMENTED
**Gate:** Passes when `DiagnosticsInstrumenter.capture_step()` in `model_scientist/diagnostics/instrumenter.py` additionally captures GPU kernel timing via `torch.profiler`, and `DiagnosticsReport` in `model_scientist/diagnostics/schemas.py` includes a `kernel_profiling: list[OperationProfile]` field containing at least 15 entries sorted by cumulative GPU time. Each `OperationProfile` must include: `op_name`, `gpu_time_us`, `cpu_time_us`, `memory_read_bytes`, `memory_write_bytes`, `bandwidth_utilization` (fraction of theoretical peak), `sm_occupancy` (fraction), and `is_fuseable` (boolean heuristic). Profiling must not increase training step time by more than 5%.

| Agent | Task |
|-------|------|
| **GPUProfilingInstrumenter** | Extend `DiagnosticsInstrumenter` in `model_scientist/diagnostics/instrumenter.py` to run `torch.profiler.profile(activities=[ProfilerActivity.CUDA], record_shapes=True, with_stack=True)` during the diagnostic capture interval. Parse the profiler's `key_averages()` output to extract per-operation GPU time, CPU time, and CUDA memory events. Map each operation to its call site in `train.py` using the stack trace. Store raw profiler output in memory during capture, summarize to `OperationProfile` dataclass at report generation time. Ensure profiling is disabled during non-diagnostic steps to avoid overhead. |
| **BandwidthUtilizationCalculator** | For each profiled operation, compute memory bandwidth utilization: `bandwidth_utilization = (memory_read_bytes + memory_write_bytes) / (gpu_time_seconds × hardware_peak_bandwidth_bytes_per_sec)`. Detect hardware peak bandwidth from `torch.cuda.get_device_properties()` — A100 SXM4 = 2039 GB/s, H100 SXM5 = 3352 GB/s, RTX 4090 = 1008 GB/s. Store the detected hardware profile in `DiagnosticsReport` under a new `hardware_profile` field: `{gpu_name, compute_capability, peak_bandwidth_gbps, sm_count, max_shared_memory_bytes}`. Operations with `bandwidth_utilization < 0.3` AND `gpu_time_us > 100` are flagged as kernel optimization candidates. |
| **FuseableOperationDetector** | Analyze the profiler trace to identify groups of consecutive operations that can potentially be fused into a single Triton kernel. Heuristics: (1) consecutive elementwise operations on same-shaped tensors → fuseable, (2) linear → activation → dropout sequences → fuseable, (3) attention score computation → mask → softmax → value projection → fuseable if not already using FlashAttention, (4) optimizer step components (gradient scaling, momentum, weight decay, parameter update) → fuseable. Output: add `is_fuseable: bool` and `fuseable_group_id: str` fields to each `OperationProfile`. Group IDs link operations that should be fused together. Store fuseable groups in a new `fuseable_groups: list[FuseableGroup]` field on `DiagnosticsReport` where `FuseableGroup = {group_id, op_names: list, combined_gpu_time_us, estimated_fused_time_us, estimated_speedup_ratio, fusion_type: str}`. |
| **DiagnosticsSchemaExtender** | Extend `DiagnosticsReport` in `model_scientist/diagnostics/schemas.py` with three new fields: `kernel_profiling: list[OperationProfile]`, `fuseable_groups: list[FuseableGroup]`, `hardware_profile: HardwareProfile`. Add corresponding `to_dict()`/`from_dict()` serialization. Update `DiagnosticsValidator` in `model_scientist/diagnostics/validator.py` to validate the new fields: check that `bandwidth_utilization` is in [0, 1], `sm_occupancy` is in [0, 1], `gpu_time_us >= 0`, and `fuseable_group_id` references exist. Add validation rule: at least 5 operations must be profiled per report, and the total GPU time across all operations must be within 20% of the wall-clock step time (sanity check against profiler overhead). |

### Subphase 1.2: Reference Implementation Catalog

**Status:** NOT IMPLEMENTED
**Gate:** Passes when `reference_catalog/` contains reference implementations for at least 5 fuseable operation groups identified by `FuseableOperationDetector` from a real `DiagnosticsReport`, each with: PyTorch code extracted from `train.py` (exact lines, not approximations), input/output tensor shape specifications derived from actual profiler trace shapes, numerical tolerance bounds per dtype (fp32: atol=1e-5 rtol=1e-5, fp16: atol=1e-2 rtol=1e-2, bf16: atol=1e-2 rtol=1e-2), and at least 5 test input configurations covering: batch=1 (edge case), batch=`DEVICE_BATCH_SIZE` (typical), batch=`DEVICE_BATCH_SIZE×2` (stress), sequence_length=1 (edge), and the model's actual sequence length from `train.py` constants.

| Agent | Task |
|-------|------|
| **ReferenceCodeExtractor** | For each `FuseableGroup` in the latest `DiagnosticsReport.fuseable_groups`, trace the group's `op_names` back to their source locations in `train.py` using the profiler stack traces stored in `OperationProfile`. Extract the exact PyTorch code path as a standalone callable function that takes input tensors and returns output tensors. Store in `reference_catalog/{group_id}/reference.py`. Each reference function must be self-contained (no `train.py` imports except `torch` and standard libs), include type annotations for all arguments and return values, and include a docstring specifying the operation's role in the training loop (e.g., "Fused RMSNorm + residual connection applied after each transformer block"). Verify each reference function produces identical output to the in-context `train.py` execution by running both on 3 test inputs and comparing with `torch.allclose()`. |
| **TensorShapeDocumenter** | For each reference implementation, document the complete tensor shape contract from the profiler's `record_shapes=True` data. Create `reference_catalog/{group_id}/shapes.json` containing: `{inputs: [{name, shape, dtype, requires_grad, stride, is_contiguous}], outputs: [{name, shape, dtype}], intermediate_buffers: [{name, shape, dtype}]}`. Include the model's actual runtime shapes from `train.py` constants: `DEPTH=8`, `n_embd=768`, `n_head` (derived), `DEVICE_BATCH_SIZE`, sequence length from the sliding window config. These shapes are critical for kernel correctness testing — a kernel that passes on small shapes but fails on actual shapes is useless. |
| **ToleranceBoundCalibrator** | For each reference implementation and each dtype (fp32, fp16, bf16), empirically determine the tightest tolerance bounds that the PyTorch reference itself satisfies under re-ordering of operations. Method: run the reference function 100 times on the same input with `torch.use_deterministic_algorithms(False)` and measure the max absolute and relative deviation across runs. Set `atol = 2 × max_abs_deviation` and `rtol = 2 × max_rel_deviation` as the correctness bounds. If PyTorch itself is non-deterministic beyond a threshold (common for fp16 reductions), document this as a known variance source. Store bounds in `reference_catalog/{group_id}/tolerances.json`: `{fp32: {atol, rtol}, fp16: {atol, rtol}, bf16: {atol, rtol}}`. |
| **TestInputGenerator** | For each reference implementation, generate test input tensors and serialize to `reference_catalog/{group_id}/test_inputs/`. Generate 5 configurations: (1) `minimal` — batch=1, seq_len=1, all values in [−1, 1], (2) `typical` — model's actual batch/seq dimensions, normal-distributed values, (3) `stress` — 2× batch size, values in [−10, 10], (4) `edge_zeros` — all-zero inputs (tests division-by-zero in normalization), (5) `edge_extremes` — values at dtype min/max (tests overflow in accumulation). For each configuration, generate both the input tensors and the expected output from the PyTorch reference. Store as `{config_name}_inputs.pt` and `{config_name}_expected.pt` using `torch.save()`. Include a `manifest.json` listing all configurations with their shapes and dtypes. |

### Subphase 1.3: Correctness Verification Engine

**Status:** NOT IMPLEMENTED
**Gate:** Passes when `KernelCorrectnessVerifier.verify()` correctly returns `CorrectnessResult(passed=True, ...)` for 3 known-correct Triton kernels (a fused GELU+bias, a fused RMSNorm, and a simple vector add) and returns `CorrectnessResult(passed=False, failure_details=...)` for 2 intentionally-broken kernels — one with an off-by-one block index (produces wrong values at block boundaries) and one with fp16 accumulation instead of fp32 (produces small but systematic numerical drift). The `failure_details` must identify: which test configuration failed, the maximum absolute error, the tensor position of the maximum error, and the dtype. Additionally, `TrainingDivergenceDetector.check()` must correctly flag the fp16-accumulation kernel as "training-divergent" after 100 steps (loss diverges by > 0.1% from reference).

| Agent | Task |
|-------|------|
| **KernelCorrectnessVerifier** | Build the core verification engine in `gpu_kernels/verification/correctness.py`. Class `KernelCorrectnessVerifier` with method `verify(kernel_callable, reference_group_id, dtypes=["fp32", "fp16", "bf16"]) → CorrectnessResult`. For each dtype and each test configuration in `reference_catalog/{group_id}/test_inputs/`: (1) load input tensors, cast to dtype, (2) run PyTorch reference, (3) run Triton kernel candidate, (4) compare outputs with `torch.allclose(atol, rtol)` from `tolerances.json`, (5) if mismatch, compute element-wise absolute error, find the maximum error position with `torch.argmax(abs_error)`, record the error magnitude and position. `CorrectnessResult` dataclass: `{passed: bool, tested_configs: int, passed_configs: int, failed_configs: list[{config_name, dtype, max_abs_error, max_rel_error, error_position: tuple, expected_value, actual_value}], total_time_seconds}`. The verifier must handle kernel crashes gracefully — if the Triton kernel raises an exception, record it as a failure with `failure_type: "crash"` rather than propagating. |
| **NumericalStabilityProber** | Build `gpu_kernels/verification/stability.py`. Class `NumericalStabilityProber` with method `probe(kernel_callable, reference_group_id, n_runs=10) → StabilityResult`. Run the kernel `n_runs` times on identical inputs to detect non-determinism. Compute the coefficient of variation of outputs across runs. Also test with inputs at dtype extremes: `torch.finfo(dtype).max * 0.9` (near overflow), `torch.finfo(dtype).tiny` (near underflow), and denormalized values. `StabilityResult` dataclass: `{is_deterministic: bool, max_cv_across_runs: float, overflow_detected: bool, underflow_detected: bool, denormal_handling: str ("correct" | "flushed_to_zero" | "error"), extreme_input_results: list}`. Flag any kernel with `is_deterministic=False` or `overflow_detected=True` — these are disqualifying for training use. |
| **TrainingDivergenceDetector** | Build `gpu_kernels/verification/divergence.py`. Class `TrainingDivergenceDetector` with method `check(kernel_callable, reference_group_id, n_steps=100, seed=42) → DivergenceResult`. Run two parallel short training sequences using the actual `train.py` configuration: one with the PyTorch reference operation, one with the Triton kernel substituted in. Same random seed, same data. Compare: (1) loss values at every step (record max absolute divergence), (2) gradient norms at every 10th step (via `DiagnosticsInstrumenter.capture_step()`), (3) parameter values at the final step (via `torch.allclose` on all model parameters with atol=1e-3). `DivergenceResult` dataclass: `{passed: bool, max_loss_divergence: float, max_grad_norm_divergence: float, final_param_match: bool, diverged_at_step: int | None, loss_curve_reference: list[float], loss_curve_kernel: list[float]}`. A kernel passes if `max_loss_divergence < 0.001 × mean(abs(loss))` (0.1% relative). |
| **CorrectnessReportGenerator** | Build `gpu_kernels/verification/report.py`. Aggregates `CorrectnessResult`, `StabilityResult`, and `DivergenceResult` into a single `KernelVerificationReport` that is stored in `gpu_kernels/reports/{kernel_id}_verification.json` and also logged to `hypothesis_journal.jsonl` via `JournalWriter` with tags `["source:kernel", "verification_report"]`. The report includes a final `verdict: "PASS" | "FAIL" | "CONDITIONAL_PASS"` where CONDITIONAL_PASS means: correctness passed, stability passed, but divergence check showed marginal drift (0.05% < divergence < 0.1%) — acceptable for short training but should be monitored in long runs via `RuntimeCorrectnessMonitor` (Phase 4). |

### Subphase 1.4: Performance Benchmarking Harness

**Status:** NOT IMPLEMENTED
**Gate:** Passes when `KernelBenchmarker.benchmark()` produces timing measurements for both reference and candidate kernels with coefficient of variation < 5% across 5 independent runs (each run = 50 warmup + 200 timed iterations), and correctly identifies a simple fused elementwise kernel (GELU + bias in one pass) as faster than the sequential PyTorch path with a measured speedup > 1.1x. The `TrainingThroughputEstimator` must produce an estimated end-to-end training throughput improvement (tok/sec delta) that is within 20% of the actual measured throughput change when the kernel is integrated into `train.py`.

| Agent | Task |
|-------|------|
| **KernelBenchmarker** | Build `gpu_kernels/benchmarking/benchmarker.py`. Class `KernelBenchmarker` with method `benchmark(kernel_callable, reference_callable, input_configs, n_warmup=50, n_timed=200, n_runs=5) → BenchmarkResult`. For each input configuration and each run: (1) run warmup iterations (discard), (2) time `n_timed` iterations using `torch.cuda.Event` with `enable_timing=True`, (3) record per-iteration times. Compute: `median_time_us`, `mean_time_us`, `std_time_us`, `cv` (coefficient of variation), `p99_time_us`. Compare reference vs. kernel: `speedup_ratio = reference_median / kernel_median`. `BenchmarkResult` dataclass: `{kernel_id, reference_group_id, input_configs_tested: int, results: list[{config_name, reference_median_us, kernel_median_us, speedup_ratio, cv_reference, cv_kernel}], overall_speedup: float (geometric mean across configs), peak_memory_kernel_bytes, peak_memory_reference_bytes, memory_savings_ratio}`. Reject benchmark results where `cv > 0.05` for either reference or kernel — rerun with more iterations or investigate system noise. |
| **MemoryBandwidthProfiler** | Build `gpu_kernels/benchmarking/bandwidth.py`. Class `MemoryBandwidthProfiler` with method `profile(kernel_callable, input_config) → BandwidthProfile`. Use `torch.profiler` with `profile_memory=True` to measure actual memory reads and writes during kernel execution. Compute `achieved_bandwidth_gbps = (total_bytes_read + total_bytes_written) / execution_time_seconds / 1e9` and `bandwidth_efficiency = achieved_bandwidth_gbps / hardware_peak_bandwidth_gbps` (hardware peak from `DiagnosticsReport.hardware_profile`). `BandwidthProfile` dataclass: `{bytes_read, bytes_written, execution_time_us, achieved_bandwidth_gbps, bandwidth_efficiency, is_memory_bound: bool (efficiency > 0.6), is_compute_bound: bool (efficiency < 0.3)}`. Memory-bound kernels with low efficiency are strong candidates for further optimization (better memory access patterns); compute-bound kernels are closer to theoretical limits. |
| **TrainingThroughputEstimator** | Build `gpu_kernels/benchmarking/throughput.py`. Class `TrainingThroughputEstimator` with method `estimate_impact(benchmark_result, diagnostics_report) → ThroughputEstimate`. Using the kernel's `speedup_ratio` from benchmarking and the target operation's `gpu_time_us` from `DiagnosticsReport.kernel_profiling`, compute the estimated end-to-end training throughput improvement: `estimated_step_time_reduction_us = operation_gpu_time_us × (1 - 1/speedup_ratio)`, `estimated_tok_per_sec_improvement = baseline_tok_per_sec × (estimated_step_time_reduction_us / total_step_time_us)`. `ThroughputEstimate` dataclass: `{kernel_id, target_operation, operation_fraction_of_step: float, estimated_step_time_reduction_us, estimated_tok_per_sec_delta, estimated_training_time_savings_percent, confidence: str ("high" if benchmark cv < 0.02, "medium" if < 0.05, "low" otherwise)}`. This estimate is used by the surrogate model as an additional feature and by the `EvaluationScheduler` to prioritize high-impact kernel candidates. |
| **HardwareCapabilityDetector** | Build `gpu_kernels/benchmarking/hardware.py`. Class `HardwareCapabilityDetector` with method `detect() → HardwareProfile`. Auto-detect: `{gpu_name, compute_capability: tuple, peak_bandwidth_gbps, sm_count, max_shared_memory_per_block_bytes, max_threads_per_block, warp_size, max_registers_per_block, clock_rate_mhz, total_memory_bytes}`. All values from `torch.cuda.get_device_properties()` supplemented by a lookup table for peak bandwidth (not available from PyTorch API). Store in `gpu_kernels/hardware_profile.json` and also inject into `DiagnosticsReport.hardware_profile`. This profile is read by the kernel generator to select appropriate block sizes and memory access patterns for the specific GPU. |

## Phase 2: Kernel Generation for Known-Fuseable Operations

**Gate:** Phase 2 is COMPLETE when: (1) the system has generated, verified (all 3 checks: correctness, stability, divergence), and benchmarked at least 5 custom Triton kernels targeting fuseable groups identified by `FuseableOperationDetector`; (2) at least 3 of those kernels PASS verification AND achieve > 1.15x speedup over the PyTorch reference; (3) at least 1 kernel is an elementwise fusion (GELU/SiLU + bias + residual), at least 1 targets the optimizer step (`MuonAdamW` parameter update fusion), and at least 1 targets attention or normalization; (4) all passing kernels are integrated into `train.py` behind a `kernel_config.json` toggle, with PyTorch fallback paths verified functional; (5) all kernel experiments (pass and fail) are logged in `hypothesis_journal.jsonl` with `source:kernel` tags and complete verification reports. Manual review of all PASS kernel code and verification reports required.
